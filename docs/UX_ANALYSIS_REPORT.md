## 1. 执行摘要

SmartSub 是一款功能完整度相当高的本地化「音视频 → 字幕 → 翻译 → 校对 → 烧录」桌面工具。代码库显示它**刚刚经历了一轮成功的信息架构重构**（资源中心整合、启动台、任务工程化、新手引导），产品「骨架」已接近专业水准；但截图审计进一步暴露：**视觉层与状态语义层明显落后于信息架构层**——骨架是对的，皮肤和表情还没跟上。

六维评估（v1 四维基础上，依据截图审计拆分出「视觉设计」细项并新增「平台公民」维度）：

| 维度       | 评分  | 核心差距                                                                 |
| ---------- | ----- | ------------------------------------------------------------------------ |
| 信息架构   | ★★★★☆ | 骨架已好，残留双入口/术语漂移/namespace 错位                             |
| 核心任务流 | ★★★☆☆ | 并发语义混乱（取消不停、暂停不停、跨工程串扰）、批量操作语义不明         |
| 编辑器体验 | ★★☆☆☆ | 零快捷键、撤销盲区、无未保存保护、长列表性能差、**一屏仅 3 条的低密度**  |
| 视觉系统   | ★★☆☆☆ | 无品牌色、**信息密度失衡**、**操作视觉权重倒挂**、**状态无色彩语义**     |
| 文案与帮助 | ★★★☆☆ | 口语化方向正确，但术语漂移、关键数字无解释（95%、评分圆点、「0 表示」）  |
| 平台公民   | ★★★☆☆ | macOS 关窗即退杀任务、默认英文菜单、无 Dock 进度；GPU 平台分支代码已做对 |

**最值得优先投入的三件事**（详见第 8 节路线图）：

1. **修复任务执行语义**——取消/暂停真正生效、状态按工程隔离、修复「全部已处理」误判、macOS 关窗保活（过夜批量任务不被杀）。这是用户信任的根基。
2. **给校对编辑器装上「专业工具的肌肉」**——快捷键体系、可靠撤销、未保存保护、列表虚拟化 + **紧凑行密度重构**（当前一屏 3 条 → 目标 10+ 条）。校对是用户停留时间最长的界面。
3. **建立轻量设计系统**——品牌主色、状态色语义体系（等待/运行/成功/失败有色可辨）、操作权重规范（危险操作降权）、统一页面头部模式与术语表。

**本轮截图审计新增的代表性发现**（v1 代码走查未覆盖）：

- 合成页「生成视频」禁用提示与实际状态矛盾——文件已选仍提示「请先选择视频和字幕文件」，真实卡点是输出路径（已定位代码根因）`[双证]`
- 校对编辑器内残留顶层「新建任务/历史任务」切换器，深层视图保留顶层导航 `[截图]`
- 模型列表一排红色实心「删除」按钮，破坏性操作视觉权重全页最高 `[截图]`
- 模型推荐双信源冲突：Hero 推荐 large-v3-turbo，档位里 medium 又标「大多数人的最佳选择」`[截图]`
- 翻译失败 89/91 时编辑器整列红色轰炸，缺「集中处理失败」的工作流 `[截图]`
- 全应用信息密度失衡：典型页面有效内容 < 30% 视口，大面积空白未承载状态/引导 `[截图]`
- 任务页「运行日志」开场显示的是 Updater 日志——全局日志混排的可视化实锤 `[双证]`

## 2. 产品定位速写（全报告的判断基准）

> 本节是 PM 视角的精炼判断基准，不展开商业分析；后文所有优先级判定均回溯到这里。

### 2.1 产品本质与差异化

一句话定位：**把「视频 → 双语字幕」的专业生产线装进个人电脑——本地、免费、批量。**

相对三类替代品的差异化（详见附录 C 竞品基准）：

- vs 专业字幕工具（Aegisub/Subtitle Edit）：**好上手**——转写+翻译开箱即用，不需要懂时间轴和插件；
- vs 剪辑软件（CapCut/剪映）：**懂字幕**——批量、双语、多格式导出，且不收云端订阅费；
- vs 在线服务：**本地隐私 + 零边际成本**——素材不出机器，长视频不按分钟计费。

这个定位的含义：SmartSub 的体验竞争力不在「功能多」，而在「**生产线可信赖 + 校对效率高**」。

### 2.2 三类核心用户与体验诉求

| 画像                   | 频率 | 典型旅程                          | 体验诉求排序                                    | 当前满足度                 |
| ---------------------- | ---- | --------------------------------- | ----------------------------------------------- | -------------------------- |
| **内容学习者**（小白） | 低频 | 拖入课程视频 → 等待 → 拿字幕      | 别让我学概念 > 快 > 好看                        | 较好（引导/推荐已做）      |
| **字幕组/翻译工作者**  | 高频 | 批量导入 → 挂机 → 逐条精校 → 交付 | 效率（键盘）> 可控（取消/撤销）> 可信（不丢稿） | **差**（本报告主要矛盾）   |
| **出海/双语创作者**    | 中频 | 自己的视频 → 双语字幕 → 烧录发布  | 风格所见即所得 > 流程顺滑                       | 中（预览是假的、字体错配） |

**洞察**：上一轮 IA 重构主要服务了画像一（新手友好），本轮应把重心移向画像二——他们是日活贡献者与口碑放大器，而当前编辑器体验（零快捷键、撤销盲区、一屏 3 条）对他们接近不可用。

### 2.3 体验北极星指标

1. **新手首胜时间**：首次启动 → 拿到第一份可用字幕 ≤ 10 分钟（含模型下载的等待体验）；
2. **校对效率**：校对 1 小时视频的纯操作耗时，目标「键盘不离手」可完成全程；
3. **信任零事故**：任务控制（取消/暂停/完成）行为与文案 100% 一致；校对永不丢稿。

### 2.4 优先级判据

**信任 > 效率 > 美感。** 工具产品信任一旦破产（点了取消还在跑、说已完成其实失败、保存覆盖了原文件），用户直接卸载，不会给美化迭代留机会。本报告 P0 全部按此判据收录。

## 3. 用户旅程与关键断点

### 3.1 新手首次旅程：「把一个视频做成双语字幕」

（🔴 阻断 / 🟡 摩擦 / 🟢 顺畅）

| 步骤              | 现状                                                                                                                                | 评级 |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ---- |
| 1. 首次启动       | 自动弹新手引导，按内存推荐模型，后台下载                                                                                            | 🟢   |
| 2. 配置翻译服务   | 引导跳到资源中心，但 17 个服务商平铺，新手不知道选哪个；表单术语专业（structuredOutput、batchSize）                                 | 🟡   |
| 3. 回到启动台     | 引导「继续」胶囊可恢复；横幅提示缺什么                                                                                              | 🟢   |
| 4. 拖视频到卡片   | 直接建工程并跳转，体验出色                                                                                                          | 🟢   |
| 5. 确认配置       | 配置条只显示常用 4-5 项，合理；但翻译服务下拉不过滤未配置项，选错要到运行时才报错                                                   | 🟡   |
| 6. 开始任务       | 阶段链 + 进度条 + 日志面板，反馈充分；但无预计剩余时间；日志面板开场显示 Updater 日志 `[截图]`                                      | 🟢   |
| 7. 中途想取消     | **点取消后正在转写的文件继续跑，spinner 不停**，用户困惑甚至以为软件卡死                                                            | 🔴   |
| 8. 全部失败后重试 | **「开始任务」被误判为已处理而拒绝执行**（truthy bug），只能逐行重试                                                                | 🔴   |
| 9. 完成 → 校对    | 横幅「去校对」只作用于第一个文件，批量场景语义不明                                                                                  | 🟡   |
| 10. 校对编辑      | 无 Ctrl+S/Ctrl+Z 快捷键；改完不切行直接保存则无法撤销；返回不提示未保存                                                             | 🔴   |
| 11. 合成到视频    | 衔接预填顺畅；但预览是假字幕；点视频 X 会把字幕也清掉（bug）；**输出路径未选时提示却说「请先选择视频和字幕文件」**；无法取消 ffmpeg | 🔴   |

### 3.2 专业用户批量旅程：「字幕组周更 20 集」（本轮新增）

| 步骤                  | 现状                                                                                 | 评级 |
| --------------------- | ------------------------------------------------------------------------------------ | ---- |
| 1. 批量导入 20 个视频 | 可多选/拖入；但重复导入不去重，无总时长/总大小预估                                   | 🟡   |
| 2. 挂机过夜           | **macOS 关窗即退出，任务全部终止**；无托盘、无 Dock 进度、无防睡眠提示               | 🔴   |
| 3. 发现配错想停       | 取消只清队列，执行中文件继续跑；跨工程状态串扰可能误杀别的工程                       | 🔴   |
| 4. 批量进入校对       | CompletionBanner 只服务第一个文件；校对页要逐个重新建任务，两边数据模型不通          | 🟡   |
| 5. 精校 2000 条字幕   | 无虚拟化卡顿 + 零快捷键 + 一屏仅 3 条 + 时间轴不可编辑——四重打击，效率远低于 Aegisub | 🔴   |
| 6. 交付导出           | 保存直接覆盖原文件无备份；「标记完成」不含保存语义，极易误以为已存                   | 🔴   |

**洞察**：两条旅程叠加后，断点高度集中在「**任务执行控制**」与「**编辑器**」——前半程（获取→配置→开始）经过重构已经很顺，后半程恰是专业用户每天的高频路径。这与第 2.2 节「重心应移向画像二」的判断互相印证。

## 4. 全局体验层分析

### 4.1 信息架构与导航

**做得好的**：

- 五项一级导航以任务动词组织，无多余层级；资源中心二级 Tabs 状态写入 URL query（`resources.tsx` L29-35），可链接、可返回。
- 各处「去下载模型 / 去配置翻译服务」的就地引导链接（`InlineConfigBar.tsx` L97-102、L157-168）形成了良好的配置闭环。

**问题**：

1. **同物异名割裂导航认知** `[双证]`：侧边栏叫「视频合字幕」（`common.json` L134），启动台卡片叫「合成到视频」（`launchpad.json` L13）；侧边栏「字幕校对」vs 卡片「校对字幕」。截图中两处并排可见，用户会怀疑是不是两个功能。
2. **「任务」一词三义** `[双证]`：一级导航「任务」、校对模块内部也叫「新建任务/历史任务/保存任务」（`profread.png`）、任务页底部「N 个任务」实为文件数。同一个词在三个上下文里指三种东西。
3. **校对/合成与任务流的关系不清** `[代码]`：校对既是独立页面又内嵌在任务页（`tasks/[type].tsx` L290-300 整页替换为 ProofreadEditor），两条路径数据模型不同（任务 IFiles vs 校对 PendingFile），任务内校对后回独立校对页找不到记录。
4. **导航高亮用子串匹配**（`Layout.tsx` L65 `p.includes('home')`）`[代码]`：query 含关键词会误判，应改路径段精确匹配。
5. **模型路径配置双入口** `[双证]`：设置页（`settings.tsx` L427-456，`setting.png` 可见）与模型 Tab（`ModelsTab.tsx` L589-600，`resourceModels.png` 可见）都能改模型存储路径，两处互不感知。建议设置页只留链接。
6. **最近任务硬截断 8 条**（`home.tsx` L173）且无「查看全部」、无筛选搜索 `[代码]`。

### 4.2 交互模式一致性

**4.2.1 危险操作确认策略不一致** `[双证]`

| 操作 | 有确认                                      | 无确认                                                                                                                                                                                                     |
| ---- | ------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 删除 | 首页删工程（AlertDialog）、删模型、删加速包 | **任务页清空列表**（清空即删工程，`tasks/[type].tsx` L191-194）、**删自定义服务商**（`ProvidersTab.tsx` L171-175，confirmDeleteMessage 是死键）、**校对列表删行/重置**（`ProofreadFileList.tsx` L588-595） |

统一策略：不可恢复的删除一律二次确认，或提供 5 秒撤销 toast（推荐后者为长期方向，不打断心流）。

**4.2.2 保存语义三套心智模型** `[双证]`（本轮新增发现）

| 界面       | 模式                                            | 证据                                     |
| ---------- | ----------------------------------------------- | ---------------------------------------- |
| 任务页     | 配置改动自动持久化，无保存按钮                  | `useFormConfig.tsx`                      |
| 校对列表   | 手动「保存任务」深色主按钮                      | `profread.png` 右上角                    |
| 校对编辑器 | 「保存」+「标记完成」双按钮，且标记完成不含保存 | `profDetail.png`、`proofread.tsx` L79-96 |

同一应用内「我的修改存了吗」有三种答案，是丢稿焦虑的根源。建议统一为「自动保存 + 显式状态指示（已保存/未保存圆点）」，「标记完成」隐含保存。

**4.2.3 即改即存的反馈节奏失控** `[代码]`：设置页 VAD 数字输入把保存绑在 onChange（`settings.tsx` L284-302），输入「250」触发 3 次保存 + 3 次 toast；ProvidersTab 每字符一次 IPC（L96-107）。应 debounce 500ms + 失焦提交；**成功静默（行内对勾微反馈），失败才打扰**（Notion/Linear 模式）。

**4.2.4 运行中的破坏性操作未防护** `[代码]`：任务运行中仍可删行（`TaskRowList.tsx` 行内 X 不受 `queueBusy` 约束）、可清空列表；删除后主进程继续处理该文件。

**4.2.5 CompletionBanner 语义不明** `[代码]`：cancelled 状态也显示、全部操作只针对 `doneFiles[0]`（`CompletionBanner.tsx` L67-86）。单文件→直接操作；多文件→下拉选择或批量视图。

### 4.3 键盘与可访问性

**这是与「专业工具」定位差距最大的一项：全应用除重命名的 Enter/Escape 外没有任何键盘快捷键。** `[代码]`

必备清单（参照 Aegisub/Subtitle Edit/CapCut 惯例）：

| 范围       | 快捷键                   | 动作                         |
| ---------- | ------------------------ | ---------------------------- |
| 全局       | Cmd/Ctrl+O               | 导入文件                     |
| 全局       | Cmd/Ctrl+,               | 打开设置                     |
| 任务页     | Cmd/Ctrl+Enter           | 开始任务                     |
| 任务页     | Delete/Backspace         | 删除选中行（需先支持行选中） |
| 校对编辑器 | Cmd/Ctrl+S               | 保存                         |
| 校对编辑器 | Cmd/Ctrl+Z / Shift+Cmd+Z | 撤销/重做                    |
| 校对编辑器 | Space（非输入态）        | 播放/暂停                    |
| 校对编辑器 | ↑/↓ 或 J/K               | 上一条/下一条字幕            |
| 校对编辑器 | Tab / Shift+Tab          | 原文⇄译文切换焦点            |
| 校对编辑器 | Cmd/Ctrl+F               | 搜索替换                     |

可访问性硬伤 `[代码]`：

## 5. 视觉设计审计（截图驱动 · 本轮新增）

> 总评：当前视觉是「shadcn/ui 默认主题 + 业务堆叠」的素颜状态——组件本身规范（间距、圆角、表单可用性都合格），但**缺一层产品级视觉决策**：没有品牌色、没有状态色语义、没有密度策略、没有操作权重规范。好消息是骨架规整，做视觉升级不需要重写界面，定义好 token 与规范后逐步替换即可。

### 5.1 布局与信息密度

**问题：全应用密度策略单一，「少内容页面」与「高频工作页面」用同一种松散密度。** `[截图]`

- `home.png`：仅 1 条最近任务时，**下半屏约 60% 视口完全空白**。启动台是默认落地页，空白未承载任何状态、引导或统计信息。
- `task.png`：文件列表是一张巨大白卡，1 个文件时 **90% 卡面空白**；卡片不随内容收缩也不利用剩余空间展示队列统计/总进度。
- `profread.png`：一行数据下方整屏空白。
- `profDetail.png`：**校对编辑器一屏只能看到约 3 条字幕**（每条 = 行头 + 两个全宽 textarea 堆叠）。91 条要翻 30 屏；2000 条的专业场景完全不可用。对比 Aegisub/Subtitle Edit 的网格视图（一屏 20-30 条）差一个数量级。校对是「扫读为主、精改为辅」的工作流，应默认紧凑单行、当前行才展开编辑态（见 9.3 线框）。
- 反例（密度合适）：`resourceModels.png` 模型列表、`setting.png` 卡片流。

**建议**：建立两档密度规范——**工作密度**（任务列表/校对列表/最近任务：紧凑行高、信息并排）与**展示密度**（启动台卡片/资源中心卡片：现状即可）；空白页面用「空态四要素」（插图+主文案+副文案+主操作）或状态摘要填充。

### 5.2 视觉层级与操作权重

**问题 1：破坏性操作视觉权重倒挂。** `[截图]`

- `resourceModels.png`：每个已安装模型行尾都是**红色实心「删除」按钮**，整页扫视时最醒目的元素是一排红删除；而真正的主操作「下载」反而是普通描边按钮。破坏性操作应是最低调的（ghost/图标态，hover 显现），主操作才该吃视觉权重。
- `profread.png`：红色垃圾桶图标直接裸露在行尾（任务页同类操作是 hover 才出现）——同类操作两种规格。

**问题 2：高频动作权重低于低频动作。** `[截图]`

- `profDetail.png`：「保存」是描边次级按钮，「标记完成」是深色主按钮。校对过程中保存是高频动作、标记完成是终态动作，权重应互换（或自动保存后弱化保存钮）。
- `profDetail.png`：撤销/重做是工具栏最左的小图标，置灰态与可用态区分微弱。

**问题 3：主按钮无品牌识别。** `[双证]` `globals.css` 是 shadcn 默认 slate 主题，primary 是近黑深蓝（`222.2 47.4% 11.2%`）。截图中所有主按钮（开始任务/校对/保存任务/一键下载）都是黑块——功能上可用，但「全应用最重要的按钮长得和遮罩一样」，没有品牌记忆点。

### 5.3 色彩系统与状态语义

**问题 1：状态没有色彩语义体系。** `[截图]`

- `home.png` 最近任务「等待中」是纯灰文本+灰点；`profread.png`「待校对」「0/1 已完成」是灰底徽章。等待/运行/成功/失败在列表里几乎同色，扫视无法分流。

**问题 2：警示色滥用与轰炸。** `[截图]`

- `profDetail.png`：翻译失败 89/91 时**右列几乎全红**（红底行 + 红框输入 + 红字提示 × 89）。警示色一旦铺满就失去警示意义，且制造焦虑。建议：失败行用左边缘红色细条 + 行头红点标记，输入框保持中性；顶部失败横幅升级为「集中处理模式」入口（只看失败项 + 逐条跳转 + 批量重试）。

**问题 3：偶发彩色与全局黑白灰断裂。** `[截图]`

- `merge.png` 顶部两张**绿色边框文件卡**是全应用唯一的大面积绿色（其余界面成功态都不用绿）；`home.png` 启动台 5 卡的 indigo/sky/emerald/amber/rose 彩色图标是全应用最有设计感的部分，但出了启动台再无呼应。

## 8. 重构路线图

按「先止血 → 再提效 → 后塑形」三阶段推进，每阶段可独立发版。

### Phase 1 「信任修复」（约 1.5-2 周）

> 目标：核心流程没有谎言——按钮说什么就做什么，挂机任务不丢。

- 修复 P0 全部 10 项（多为局部修改，互相独立）。
- 最大项是任务执行语义重构：队列携带 projectId、cancel 发真实中断信号（whisper addon abort + 翻译 cancellation token）、日志按工程过滤、UI 文案明确「停止派发 vs 中断当前」。
- macOS 关窗保活 + Dock 进度条。
- 统一危险操作：抽 `useConfirmOrUndo` hook（AlertDialog 或 5 秒撤销 toast），替换全部裸删除（P1#22 顺带完成）。
- 保存反馈降噪：debounce + 失焦提交，成功静默失败 toast（P1#23）。
- 合成页提示矛盾修复 + 输出路径默认值（P1#13，改动极小收益大）。

**验收标准**：① 任务运行中点「取消」，正在转写的文件 5 秒内真正停止；② 全部失败的列表可一键重跑；③ A 工程运行时 B 工程页面无 A 的控制按钮与日志；④ macOS 关窗后任务继续、Dock 显示进度；⑤ 校对编辑任意修改后退出必有保存提示，保存生成 .bak；⑥ 合成页两文件已选时按钮可用或提示准确指向缺失项。

### Phase 2 「编辑器专业化」（约 2-3 周）

> 目标：校对编辑器达到「专业字幕工具」基线，留住画像二用户。是 2.18 的主题版本（「编辑器大版本」是好的对外叙事）。

- 快捷键体系（4.3 清单）+ 帮助菜单「快捷键速查」面板 + 底部快捷键提示条。
- 撤销系统重构为命令模式（行级 diff），覆盖逐字编辑；dirty 状态 + 离开保护 + 保存备份。
- **列表重构（密度+性能一体）**：紧凑单行模式（原文|译文并排预览）+ 当前行展开编辑 + `@tanstack/react-virtual` 虚拟化 + 播放联动二分索引 + 行级 memo（见 9.3 线框）。
- 编辑态隐藏顶层切换器（P1#14）；「集中处理失败」模式：失败筛选 + 批量重试 + 处理进度（P1#15）。
- 时间轴行内编辑（点击时间出现 mm:ss,ms 输入，校验区间重叠）。
- 合并改列表多选（Shift+点选）；搜索替换加逐条确认与高亮。
- 批量 AI 优化可取消（IPC abort 通道）；AI 优化支持纯转写模式。
- 合成页真实字幕预览（复用解析层）+ ffmpeg 取消 + 软字幕封装选项 + 中文默认字体修正。

**验收标准**：① 2000 条字幕滚动 60fps、输入无可感延迟；② 全程仅用键盘完成「打开→定位失败→修改→保存→标记完成」；③ 任意编辑可撤销（含保存前最后一次输入）；④ 一屏可见 ≥10 条字幕；⑤ 合成预览与 ffmpeg 成品字体/位置基本一致。

### Phase 3 「品牌与一致性」（约 1.5-2 周 + 持续）

> 目标：从「能用的工具」到「有性格的产品」。

- 设计 token 升级：品牌主色（建议 indigo 域 oklch）+ 四态状态色（waiting/running/success/failed）+ focus ring 统一；操作权重规范（危险操作降权为 ghost/hover 显现）；移除紧凑模式 hack；补齐暗色破损。
- 页面头部统一为「枢纽页/工作页」两模板（5.4）；空态四要素模板；状态徽章彩色化。
- 术语表落地（4.4 表格）：一次性全量替换 + 贡献者文档；namespace 重组（拆 proofread/editor，清死键）；CI 校验 zh/en key 对等。

- **对外形象**：README/官网全套新截图 + IA 重构 before/after release note。

**验收标准**：① 全应用主按钮/激活态/运行态出现品牌色，删除按钮无大面积红；② 全部页面头部收敛为 2 种模板；③ zh/en key 对等 CI 绿；④ 术语表全量替换无「听写/识别」残留；⑤ README 首屏是 v2.17+ 新界面。

### 节奏建议

P0（Phase 1）随 2.17 正式版发布；Phase 2 是 2.18 主题；Phase 3 伴随 2.18-2.19 持续落地。每项改动配 before/after 截图进 Changelog。**具体的版本切分、周节奏与风险预案见第 11 章。**

## 10. 设计原则与发版自检清单

### 10.1 设计原则（建议写入 CONTRIBUTING）

原 5 条保留，新增 2 条视觉原则（6、7）：

1. **按钮即契约**：控件文案必须与系统实际行为一致。「取消」就要停，「已完成」就要真完成。做不到的能力，文案写保守（「停止派发新任务」）。
2. **渐进披露，但不藏断点**：常用项表层、专业项折叠（InlineConfigBar/AdvancedSheet 模式已验证），但「下一步要什么」的前置条件必须表层可见（缺模型/缺服务横幅模式已验证，推广到所有模块——包括「缺输出路径」）。
3. **破坏不可逆，必须可反悔**：删除/覆盖/清空，要么二次确认，要么 5 秒撤销，要么留备份。三者必居其一。
4. **成功安静，失败大声**：自动保存成功不打扰（行内微反馈），失败显式可见且可重试。Toast 是稀缺资源。
5. **键盘是专业用户的母语**：高频界面每个核心动作都有快捷键，并在 tooltip 标注。
6. **密度服务于任务**：浏览型界面可以松，工作型界面（列表、编辑器）必须紧——衡量标准是「一屏能承载多少次决策」。
7. **色彩即状态语义**：色彩优先留给状态（运行/成功/失败/警告）与品牌主操作；装饰性用色克制。警示色一旦铺满即失效。

### 10.2 发版前体验自检清单（每个 release 跑一遍）

- [ ] 新增/修改的每个按钮，行为与文案一致（原则 1）

## 11. 研发执行节奏与版本迭代计划

> 适配单人维护者 + AI 协作的产能模型：每周按 4 个有效开发日估算，留 1 天给测试、发版与社区。所有日程含义是「专注投入的工作周」，可按实际可支配时间等比拉伸。

### 11.1 执行原则

1. **每个版本独立交付价值**：任何时点停下来，已发布的版本都是完整可用的——不存在「半成品状态等下个版本补齐」。
2. **高风险重构不同版本叠加**：任务执行语义重构（队列/中断信号）与编辑器重构（撤销/虚拟化）是两块最大的高风险改动，分属两个版本，避免回归排查时互相污染。
3. **范围冻结**：版本开工后新想法一律进 backlog；只有新发现的 P0 级缺陷允许插队。
4. **门禁前置**：第一周先搭好 i18n key 对等 CI 与冒烟清单，之后每次合并都有门禁；每个版本发布前过 10.2 自检清单。
5. **trunk 永远可发布**：大块改动走 feature 分支 + beta 通道，主干随时可打正式版。
